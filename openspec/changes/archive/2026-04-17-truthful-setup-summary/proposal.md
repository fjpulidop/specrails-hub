## Why

The post-install completion screen in the setup wizard misrepresents what was installed. It shows three tiles — `Agents`, `Personas`, `Spec` — where:
- `Spec` is a mislabeled tile whose value is actually the slash-command count.
- `Personas` shows a non-zero number after a quick install even though personas are only generated by the `/specrails:enrich` step of a full tier install — the tile is counting boilerplate template files, not project-specific personas.
- The `commands` count silently sums `.claude/commands/sr/` (deprecated prefix) and `.claude/commands/specrails/` (canonical prefix), double-counting every command that exists under both prefixes, and ignores the `.claude/commands/opsx/` commands that `specrails-core` also installs.

The result is a final screen that confuses users about which slash-command prefix to type, which personas exist, and what `specrails-core` actually put on disk. We want the screen to state the truth and nothing more.

## What Changes

- `SetupSummary` (server) is extended with:
  - `tier`: which tier the user installed (`quick` | `full`).
  - `specrailsCommands`: count of `.claude/commands/specrails/*.md`.
  - `opsxCommands`: count of `.claude/commands/opsx/*.md`.
  - `personas`: unchanged semantics (count of `.claude/agents/personas/*.md`), but only surfaced in the UI for the full tier.
  - `legacySrRemoved`: count of `.md` files that existed in `.claude/commands/sr/` and were deleted during this install run.
  - `agents`: unchanged.
- `computeSummary()` stops returning a single `commands` number. The legacy `commands` field is **BREAKING**ly replaced by `specrailsCommands` + `opsxCommands`.
- Setup manager deletes `.claude/commands/sr/` (and its contents) on every successful install / reinstall run. The deletion happens after `install.sh` runs and before `computeSummary()` reads the final tree, so the sweep is reflected in the summary.
- The completion step of `SetupWizard` (`CompleteStep`) is restructured:
  - Drops the mislabeled `Spec` tile.
  - Splits commands into two labelled tiles: `/specrails:*` and `/opsx:*`.
  - Hides the `Personas` tile on quick tier; shows it on full tier.
  - Shows a one-line notice when `legacySrRemoved > 0`: "Removed N legacy `/sr:*` commands".
- No changes to `install.sh`, `deployTemplates`, the install wizard flow, `specrails-core` templates, or the command surface area. The only install-side change is the `/sr:*` cleanup sweep inside `setup-manager`.

## Impact

- **Server**: `server/setup-manager.ts`
  - `SetupSummary` interface shape changes (BREAKING within the project — this type is consumed only by the server itself and by the client's wizard message handling).
  - `computeSummary()` rewritten to produce the new fields.
  - New small helper that removes `.claude/commands/sr/` before `computeSummary()` runs, returning the number of files removed so the summary can report it.
  - Tier is already known to the server inside the install flow; it is threaded into the summary payload rather than recomputed.
- **Client**: `client/src/components/SetupWizard.tsx`
  - `SetupSummary` TypeScript type mirrors the server shape.
  - `CompleteStep` renders the new tile layout, the tier-conditional personas tile, and the legacy cleanup notice.
  - Fallback defaults (`{ agents: 0, personas: 0, commands: 0 }`) used on malformed messages become `{ agents: 0, specrailsCommands: 0, opsxCommands: 0, personas: 0, legacySrRemoved: 0, tier: 'quick' }`.
- **WebSocket protocol**: the `summary` payload carried on `setup.complete` messages changes shape. No other consumer exists; both sides are updated atomically.
- **Tests**: `server/setup-manager.test.ts` and `client/src/components/__tests__/SetupWizard.test.tsx` need updates for the new summary shape and UI.
- **No impact** on: `install.sh`, `deployTemplates`, specrails-core templates, job queue, chat manager, or any other project-scoped state.
