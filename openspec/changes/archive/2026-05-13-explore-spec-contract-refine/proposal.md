## Why

Spec drafts produced by Explore Spec are often correct but too brief, so downstream agents (Architect → Developer → Reviewer) reinvent decisions — enum names, file paths, function signatures, edge cases. The fix is not "longer specs" (more text = more interpretation) but a **Contract Layer**: an explicit, prescriptive block of anti-reinvention anchors (exact identifiers, data shapes, state machines, invariants, file touch list) generated by a dedicated **refinement turn** after the user-facing draft is final.

## What Changes

- Add a "Contract Layer" section to every Explore Spec draft, structured as: Naming Contract, Data Shapes, State Machine (when applicable), Invariants, File Touch List.
- Add a **refinement turn** in `ChatManager` Explore lifecycle: after the user commits the spec, run one additional Claude turn whose sole job is to fill the Contract Layer additively (never modifies user-authored sections).
- Refinement runs invisibly by default (no extra UI surface beyond a brief "Afinando contrato…" status pill, gated by `VITE_FEATURE_EXPLORE_CONTRACT_REFINE`).
- Refinement is **off-by-default** behind a per-project setting `config.explore_contract_refine_enabled` (mirrors the `explore_mcp_enabled` pattern in `queue_state`).
- Refinement uses the same model as the main Explore turn (no per-feature model override in v1).
- Refinement failure is degraded-but-functional: the ticket commits without Contract Layer and a `chat_warning` is logged. No retry.
- Quick mode (`POST /tickets/generate-spec`) is **out of scope** — Quick = fast by definition.
- `ai_invocations` records the refinement turn as `surface='explore-spec'` with `surface_ref_id` linking to the parent conversation, so spending analytics naturally absorb the extra cost.

## Impact

- **Server**: `server/chat-manager.ts` Explore lifecycle gains a `runContractRefine(conversationId)` step after `from-draft` / commit. New helper module `server/explore-contract-refine.ts` builds the refinement system prompt and parses the additive result. `server/project-router.ts` exposes `GET/PATCH /api/projects/:projectId/explore-contract-refine-enabled`. `ai-invocations` capture in refinement spawn.
- **Schema**: New key `config.explore_contract_refine_enabled` in `queue_state` (no migration — key/value table).
- **Client**: `SettingsPage` gains a toggle in the existing "Explore Spec" card. `ExploreSpecShell` shows the new status pill while refine is in flight (post-commit; non-blocking — user can already close the shell). `TicketDetailModal` renders the Contract Layer as a collapsed section by default.
- **Spec generator prompt**: Updated to declare the Contract Layer as a section that the *refinement* turn will fill, so the initial draft stops trying to inline naming/shape details and stays focused on intent.
- **Cost / latency**: ~1.5–2× cost per committed spec, +3–8 s background latency. No user-perceived wait (refine runs after commit).
- **Out of scope (future changes)**: Quick-mode contract refine; iterative refine (multi-turn until "complete"); per-feature model override (e.g., Haiku for refine); user-visible diff of refinement output.
